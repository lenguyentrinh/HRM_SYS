HRM System

Web app quản lý nhân sự nội bộ — chấm công QR, tính lương, nghỉ phép

React 18 + Vite TypeScript Strict Supabase + PostgreSQL Tailwind + shadcn/ui Edge Functions (Deno) ~50 nhân viên 2 Portal riêng biệt
1. Mô tả App

App là gì?

HRM System là web app nội bộ để quản lý nhân sự cho doanh nghiệp vừa và nhỏ (~50 người). App giải quyết 3 nghiệp vụ cốt lõi: chấm công bằng QR code, tính lương tự động và quản lý nghỉ phép.

👨‍💼 Admin Portal /admin/*

  • Dành cho super_admin và manager
  • Quản lý nhân sự, ca làm việc, chấm công
  • Duyệt đơn nghỉ phép, đổi ca
  • Tính lương hàng tháng
  • Báo cáo & thống kê

👤 Employee Portal /

  • Dành cho employee
  • Quét QR check-in/check-out hàng ngày
  • Xem lương dự kiến realtime
  • Nộp đơn xin nghỉ, yêu cầu đổi ca
  • Lịch sử chấm công cá nhân

🔑 3 Roles người dùng

  • super_admin — Toàn quyền, cấu hình hệ thống
  • manager — Quản lý vận hành, duyệt đơn, xem lương
  • employee — Chấm công, xem lương cá nhân, nộp đơn

⚙️ Đặc điểm kỹ thuật

  • Custom auth (SHA-256, không dùng Supabase Auth)
  • Multi-branch ready (branch_id trên mọi bảng)
  • QR token tự động sinh qua pg_cron
  • Realtime notifications qua Supabase Realtime
  • Lương tính bởi Edge Function (Deno)

U
Tôi muốn xây dựng một web app HRM nội bộ cho công ty ~50 nhân viên. Stack: React 18 + Vite + TypeScript + Supabase. Cần 2 portal: Admin và Employee. Auth bằng số điện thoại + mật khẩu, không dùng Supabase Auth — tự làm custom với bảng users và SHA-256 hash. Bắt đầu từ đâu?
AI
Tôi sẽ setup theo thứ tự này:

1. Khởi tạo project
  • Vite + React + TypeScript, cài Tailwind + shadcn/ui
  • TanStack Query (server state), Zustand (UI state), React Router v6
2. Custom Auth flow — vì không dùng Supabase Auth, cần:
  • Bảng users với phone, password_hash (SHA-256 hex), role

Bắt đầu với migration SQL tạo schema trước nhé?
U
Đúng rồi. Tạo migration tạo bảng users và employees. Kèm RLS policy: anon có thể đọc users để login, nhưng chỉ đọc đúng record của mình sau khi biết id.
AI
Đây là migration SQL: 
-- users table (custom auth, no supabase auth)
create table users (
  id          uuid primary key default gen_random_uuid(),
  phone       text unique not null,
  password_hash text not null,  -- SHA-256 hex
  role        text not null check (role in ('super_admin','manager','employee')),
  branch_id   uuid references branches(id),
  created_at  timestamptz default now()
);

-- RLS: permissive cho login flow (app tự validate)
alter table users enable row level security;
create policy "allow anon read for login"
  on users for select to anon using (true);

-- employees table
create table employees (
  id          uuid primary key default gen_random_uuid(),
  user_id     uuid unique references users(id),
  branch_id   uuid not null references branches(id),
  full_name   text not null,
  base_salary numeric not null default 0,
  status      text default 'active',
  ...
);
Login flow trong auth.ts: hash password bằng crypto.subtle.digest('SHA-256') (Web Crypto API, không cần thư viện), rồi supabase.from('users').select().eq('phone', phone).eq('password_hash', hash).single().
U
Làm luôn authStore.ts và route guard. Sau khi login: super_admin/manager → /admin/dashboard, employee → /.
AI
authStore.ts dùng Zustand với persist middleware vào key hrm-auth:

Route guard đọc store synchronously — không cần useEffect hay loading state khi app khởi động. Nếu user === null → redirect /login. Nếu role không khớp → redirect về portal đúng của role đó. Logic này đặt trong router.tsx dùng <Navigate> component của React Router.
📷 Phase 2 — Tính năng QR Check-in
U
Làm tính năng chấm công QR. Flow: tablet đặt tại văn phòng hiển thị mã QR của ca hiện tại, nhân viên dùng điện thoại quét. QR token được sinh tự động mỗi ngày bằng pg_cron. Mỗi ca có 1 token UUID, expire đúng giờ kết ca.
AI
Cần 3 phần làm song song:

1. Edge Function generate-qr — nhận {run_all: true}, đọc tất cả ca active, tính expires_at = end_time của ca, upsert vào bảng qr_tokens với UNIQUE (shift_id, date).

2. pg_cron job — chạy mỗi ngày lúc 06:30 VN gọi function trên.

3. TabletQRPage (/tablet/:branch_id, không cần login) — load ca đang active theo giờ thực, fetch token, render <QRCode value="https://app.com/checkin?token=UUID"/>, auto-refresh 60s.

4. CheckinPage (mobile) — bật camera, dùng BarcodeDetector API scan liên tục, khi phát hiện QR → extract token từ URL → confirm screen → gọi Edge Function checkin.

Bắt đầu với Edge Function checkin (phần validate) vì đây là logic phức tạp nhất?
U
Đúng. Validate cần: token hợp lệ + chưa hết hạn, nhân viên phải thuộc ca đó (ưu tiên shift_schedules, fallback employee_shift_assignments), chưa check-in ca này rồi. Nếu ok thì tính số phút trễ và ghi attendance_records.
AI
Logic validate trong checkin/index.ts:

ATT-01: SELECT * FROM qr_tokens WHERE token = $1 → kiểm tra is_active = true và expires_at > now()
ATT-02: Lookup ca của NV ngày đó — query shift_schedules theo employee_id + date trước. Nếu không có → query employee_shift_assignments theo employee_id + month + year. So sánh với qrToken.shift_id.
ATT-03: Query attendance_records WHERE employee_id + date + shift_id → nếu có check_in_at rồi → lỗi "đã check-in"
ATT-05: isLate = now > (shiftStart + grace_period_minutes)

Sau đó upsert attendance_records và insert notifications cho NV.
U
Nhân viên check-in sai ca thì nên có text nội dung rõ ràng, không trả lỗi từ edge function như vậy. Lỗi hiện tại chỉ hiện "Edge Function returned a non-2xx status code".
AI
Vấn đề: FunctionsHttpError.message luôn là generic string đó, message thật nằm trong error.context (Response object). Fix trong useQrCheckin:
if (error) {
  let message = error.message
  try {
    const body = await (error as { context?: Response })
      .context?.json()
    if (body?.message) message = body.message
  } catch { /* ignore */ }
  throw new Error(message)
}
Sau đó NV sẽ thấy đúng: "Bạn không thuộc ca làm việc này" thay vì lỗi kỹ thuật.

U
lỗi này khá là một vấn đề lớn vì khi mở webapp trên web mobile thì không mở được camera để quét QR

chrome vẫn không mở được máy ảnh trên Iphone

🐛 Bug – QR token báo hết hạn dù chưa hết ca
U
nhân viên ca lỡ (9h–15h) scan QR ca lỡ → báo "Mã QR đã hết hạn hoặc không hợp lệ". Tôi đính kèm curl request thực tế. Token UUID là 788cf9dc-fe86-4bc6-b4c8-c50daaf8da73.
AI
Đọc lại generate-qr/index.ts và checkin/index.ts. Tìm ra 3 bugs timezone:

Bug 1 – dateStr dùng UTC thay vì VN time:
Cron chạy 30 23 * * * UTC = 06:30 sáng VN ngày hôm sau. Nhưng new Date().toISOString() trả UTC → lúc 06:30 VN vẫn là 23:30 UTC ngày hôm trước → dateStr = "2025-05-21" (hôm qua!). Token được lưu với date = hôm qua.

Bug 2 – expires_at không có timezone:
new Date("2025-05-21T15:00:00") trên server UTC = 15:00 UTC = 22:00 VN. Nhưng ca kết thúc 15:00 VN = 08:00 UTC. Vậy expires_at của hôm qua (2025-05-21T15:00:00Z) khi so với now (sáng 22/05 UTC) → đã hết hạn!

Bug 3 – shift.start_time format trong checkin:
PostgreSQL TIME trả "09:00:00". Code append thêm :00 → "09:00:00:00" = Invalid Date → tính trễ sai.
U
Fix hết đi
AI
Fix trong generate-qr/index.ts: 
// Dùng VN date (+7h vào UTC)
const VN_OFFSET = 7 * 60 * 60 * 1000
const dateStr = body.date ??
  new Date(Date.now() + VN_OFFSET)
    .toISOString().split('T')[0]

// expires_at đúng timezone VN
const expiresAt = new Date(
  `${expireDate}T${endTimeHHMM}:00+07:00`
)
Fix trong checkin/index.ts: 
// Slice "09:00:00" → "09:00", thêm timezone VN
const startHHMM = shift.start_time.slice(0, 5)
const shiftStart = new Date(
  `${dateStr}T${startHHMM}:00+07:00`
)
Sau khi deploy, cần regenerate QR hôm nay (token cũ đã sai date).
💰 Phase 3 — Edge Function tính lương
U
Làm Edge Function calculate-payroll. Công thức:
- salary_earned = base_salary × (working_days / standard_days)
- overtime_pay = (base_salary / standard_days / shift_hours) × ot_hours × ot_multiplier
- gross = salary_earned + allowance + ot_pay + bonus - late_penalty - absent_penalty
- net = gross - bhxh - tax
Sau khi tính xong lưu vào payroll_records với status draft.
AI
Function nhận {employee_id, month, year}, đọc dữ liệu song song:
  • attendance_records trong tháng → đếm ngày công, giờ OT, số lần trễ
  • employees → base_salary, allowance, type
  • payroll_configs → ot_multiplier, late_penalty_per_minute, bhxh_rate
  • leave_requests (approved) → xác định ngày nghỉ có lương
Điểm quan trọng: standard_days là số ngày làm việc trong tháng (trừ cuối tuần + ngày lễ), không phải 26 hay 30 cố định. Tính bằng cách đếm ngày trong tháng loại trừ Thứ 7/CN và holiday_dates từ config.

Sau khi tính xong: upsert payroll_records với onConflict: 'employee_id, month, year' — cho phép tính lại nhiều lần khi vẫn còn ở status draft.
U
Thêm tính năng: admin nhấn "Xác nhận phát lương" thì status chuyển confirmed và lock lại — không tính lại được nữa trừ khi super_admin unlock.
AI
Thêm check ở đầu calculate-payroll:
const existing = await supabase
  .from('payroll_records')
  .select('status')
  .eq('employee_id', employee_id)
  .eq('month', month).eq('year', year)
  .maybeSingle()

if (existing?.status === 'confirmed') {
  return { success: false,
    message: 'Bảng lương đã được xác nhận, không thể tính lại' }
}
Thêm mutation useConfirmPayroll ở frontend: update status draft → confirmed. Chỉ super_admin mới có nút unlock (set lại về draft). Thao tác này ghi vào audit_logs.

5. Bài tập thực hành AI

🎯 Luồng: QR Check-in End-to-End

Tái hiện lại luồng chấm công QR từ đầu đến cuối — một trong những tính năng phức tạp và thú vị nhất của app, chạm đủ các tầng: Frontend → Edge Function → Database → Realtime.

⏱ ~4–6 giờ 🧩 3 components ⚡ 2 Edge Functions 🗄 4 database tables 📱 Mobile-first UI

Tổng quan luồng dữ liệu

pg_cron (06:30 VN mỗi ngày)
   ↓ gọi
Edge Function: generate-qr
   ↓ upsert token UUID vào
DB: qr_tokens (shift_id, date, token, expires_at)
   ↓ Tablet đọc token, render
TabletQRPage — hiển thị QR lớn (react-qr-code)
   ↓ NV quét bằng điện thoại (URL: /checkin?token=UUID)
CheckinPage — camera scan → parse token → confirm screen
   ↓ gọi
Edge Function: checkin
   ↓ validate: token hợp lệ? NV thuộc ca này? Đã check-in chưa?
   ↓ tính trễ: now > (shift.start_time + grace_period)?
   ↓ upsert vào
DB: attendance_records (check_in_at, status, late_minutes)
   ↓ insert vào
DB: notifications → Supabase Realtime → NV nhận ngay

Chi tiết từng bước

1

pg_cron sinh QR mỗi sáng

Job SQL chạy lúc 30 23 * * * UTC (= 06:30 VN). Gọi Edge Function generate-qr với {"run_all": true}.

  • Đọc tất cả ca active từ bảng shifts
  • Tính dateStr theo VN timezone (+7h vào UTC)
  • Tính expires_at = end_time+07:00 của ca đó
  • Upsert vào qr_tokens với constraint UNIQUE(shift_id, date)
2

Tablet hiển thị QR

Trang src/features/tablet/pages/TabletQRPage.tsx — không cần đăng nhập, URL /tablet/:branch_id.

  • Load tất cả ca đang active (isCurrentShift())
  • Nếu nhiều ca trùng giờ → dropdown chọn ca, auto-select ca vừa bắt đầu gần nhất
  • Fetch token từ qr_tokens theo shift_id + date + is_active
  • Render <QRCode value="https://app.com/checkin?token=UUID" />
  • Refresh mỗi 60s, đếm ngược thời gian hết hạn
3

Nhân viên quét QR bằng điện thoại

Trang src/features/employee-portal/pages/CheckinPage.tsx — cần đăng nhập, mobile-first.

  • BarcodeDetector API (Chrome Android/Desktop) — scan liên tục không cần chụp
  • jsQR (canvas fallback) — iOS Safari, Firefox
  • File input capture="environment" — khi getUserMedia bị từ chối
  • Parse URL từ QR data để extract token UUID
  • Hiển thị màn hình confirm: tên ca, token snippet, nút "Xác nhận Check-in"
4

Edge Function validate & ghi attendance

File supabase/functions/checkin/index.ts — chạy trên Deno, dùng service role key.

  • ATT-01: Lookup token → token tồn tại? is_active = true? expires_at > now?
  • ATT-02: Xác định ca NV hôm nay: ưu tiên shift_schedules, fallback employee_shift_assignments. So sánh với qrToken.shift_id
  • ATT-03: Đã check-in ca này rồi? → báo lỗi
  • ATT-05: Tính trễ: now > (shift.start_time + grace_period_minutes)
  • Upsert attendance_records với UNIQUE(employee_id, date, shift_id)
  • Insert notifications → Supabase Realtime push ngay cho NV
5

Kết quả hiển thị cho nhân viên

  • ✅ Thành công: "Check-in thành công lúc 07:02 – Đúng giờ ✓"
  • ⚠️ Đi trễ: "Check-in thành công lúc 07:45 – Đi trễ 45 phút"
  • ❌ Lỗi: "Bạn không thuộc ca làm việc này" / "Mã QR đã hết hạn"

🧠 Concepts cần nắm

Timezone Vietnam (UTC+7)

Server Edge Function chạy UTC. Phải cộng +7h khi tính ngày VN:

const vnDate = new Date(Date.now() + 7 * 3600000) .toISOString().split('T')[0] // expires_at đúng timezone: new Date(`2025-05-22T15:00:00+07:00`)

PostgreSQL TIME format

Cột TIME trả về "09:00:00" (có giây). Nếu append thêm :00 sẽ thành "09:00:00:00" = Invalid Date.

// ❌ Sai new Date(`2025-05-22T${shift.start_time}:00`) // ✅ Đúng const hhmm = shift.start_time.slice(0, 5) new Date(`2025-05-22T${hhmm}:00+07:00`)

Ca lookup priority

Khi cần biết ca của NV ngày X:

// 1. Override cụ thể theo ngày shift_schedules .eq('employee_id', empId) .eq('date', dateStr) // 2. Fallback: ca tháng mặc định employee_shift_assignments .eq('employee_id', empId) .eq('month', month) .eq('year', year)

BarcodeDetector + jsQR fallback

Strategy ưu tiên theo browser:

if ('BarcodeDetector' in window) { // Chrome Android, Desktop Chrome/Edge detector.detect(videoElement) } else { // iOS Safari, Firefox — jsQR canvas loop ctx.drawImage(video, 0, 0) jsQR(imageData.data, w, h) }
